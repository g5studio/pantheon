---

當用戶輸入 `resolve-conflict` 時，必須以 **Answer 視窗主導**整個流程，並嚴格遵守「先理解、再修正」原則。

## 核心原則

1. **先確認再執行**：任何會改動檔案或 Git 狀態的操作，都必須先取得用戶確認。
2. **不得擅自推測意圖**：若無法明確判定任一側改動意圖，必須立即停止並用 Answer 視窗提問。
3. **逐檔處理**：一次只處理一個衝突檔案，該檔完成並經用戶確認後，才能進到下一檔。
4. **意圖不可被破壞**：修正衝突時必須保留 current 與 incoming 的既有意圖。
5. **全流程可追溯**：每一步均需產生可回溯紀錄，最終彙整為完整合併報告。

## 流程總覽（必須先展示）

### Step 0：啟動前說明與理解確認（Answer 視窗）

啟動後，先用 Answer 視窗列出完整預期流程與注意事項，至少包含：

- 會先檢查並清空本地改動（回到衝突剛發生狀態）
- 會追溯衝突內容來源 commit 與 Jira
- 會閱讀關聯 MR 的 commit 來還原改動目的
- 每個衝突檔都會做雙方意圖比對並請用戶確認
- 不明確時會中止並詢問，不會自行推測
- 完成後會產生檔案級與全域級報告
- 最後經最終確認才會 `git merge --continue`

Answer 視窗選項必須為：

- `理解`
- `有疑問`

規則：
- 若用戶選 `有疑問`，必須先釐清問題，再重複 Step 0，直到用戶選擇 `理解`。
- 未取得 `理解` 前，禁止進入主要程序。

---

### Step 1：確認並重置本地狀態

1. 檢查目前是否有本地改動（staged / unstaged / untracked）。
2. 若有改動，先清除所有本地改動，確保回到衝突剛發生時狀態。
3. 將重置前後狀態摘要回報給用戶確認。

注意：
- 若偵測到存在使用者未備份的風險資料，必須先透過 Answer 視窗明確告知風險並取得確認。
- 未確認前不得執行清除操作。

---

### Step 2：逐一尋訪衝突檔案並追溯來源 commit

對每個衝突檔案依序執行：

1. 解析衝突區塊（current / incoming）。
2. 追溯衝突內容對應的原始 commit（雙方都要）。
3. 記錄：
   - 檔案路徑
   - 衝突區塊摘要
   - current commit 單號
   - incoming commit 單號

---

### Step 3：由 commit 追溯 Jira ticket

對每個來源 commit：

1. 解析 commit message 中的 ticket（可多張）。
2. 建立 commit -> Jira ticket 對照表（不得遺漏）。

若無法解析 ticket：
- 立即停止，透過 Answer 視窗請用戶補充。
- 禁止自行猜測 ticket。

---

### Step 4：閱讀 Jira 與關聯 MR commits，還原改動意圖

對每個 ticket：

1. 閱讀 Jira card（summary、description、comments、關聯資訊）。
2. 讀取關聯 MR 的全部 commits（非僅最新一筆）。
3. 彙整此 ticket 在該衝突區塊中的實際改動意圖。

若任一步驟無法明確判斷意圖：
- **必須停止**，用 Answer 視窗提問。
- 在用戶補充前，不得進入修正。

---

### Step 5：意圖對照確認（每檔必做，Answer 視窗）

針對當前衝突檔案，輸出以下對照資訊：

- current：
  - commit
  - ticket
  - 改動意圖
- incoming：
  - commit
  - ticket
  - 改動意圖

再用 Answer 視窗請用戶確認是否正確。

建議選項：
- `正確，可開始修正`
- `不正確，需調整分析`

規則：
- 若為 `不正確`，必須修正分析後重新確認。
- 未得到 `正確，可開始修正` 前，不可修改衝突內容。

---

### Step 6：進行衝突修正與意圖保真回歸檢查

僅在 Step 5 獲得授意後執行：

1. 實施衝突修正。
2. 完成後執行回歸檢查，確認：
   - current 意圖仍成立
   - incoming 意圖仍成立
   - 無額外引入與雙方意圖無關的邏輯改變

若檢查不通過：
- 必須回到修正步驟重新調整，直到通過。

---

### Step 7：單檔衝突改動報告（Answer 視窗確認）

每個檔案完成後，輸出簡要報告，至少包含：

- 衝突檔案
- 修正策略
- 如何保留 current 意圖
- 如何保留 incoming 意圖
- 回歸檢查結果

透過 Answer 視窗請用戶確認：
- `確認，處理下一檔`
- `有疑問，先調整`

僅在 `確認，處理下一檔` 後，才可進入下一個衝突檔案。

---

### Step 8：全部完成後的最終合併報告與 merge continue

當所有衝突檔案皆完成：

1. 產生完整衝突合併報告（全檔案彙總）。
2. 報告需包含：
   - 每個衝突檔案的處理方式
   - 每檔 current/incoming 意圖保留證據
   - 風險與驗證結果
3. 透過 Answer 視窗做最終確認。

最終確認選項：
- `確認並繼續 merge`
- `退回調整`

只有在 `確認並繼續 merge` 時，才允許：
1. 將完整報告附加進本次 merge commit description。
2. 執行 `git merge --continue` 完成衝突合併。

---

## 禁止事項

- ❌ 未經 Answer 視窗確認就直接修衝突
- ❌ 在意圖不明確時自行推測
- ❌ 跳過單檔報告與確認
- ❌ 未最終確認就 `git merge --continue`
- ❌ 只讀最新 commit 就下結論（必須看關聯 MR 全部 commits）

## 建議輸出格式（每檔）

每一檔衝突建議以固定模板回報：

1. 衝突檔案
2. Current 來源（commit / ticket / 意圖）
3. Incoming 來源（commit / ticket / 意圖）
4. 擬採修正策略
5. 保真檢查清單與結果
6. 單檔報告確認結果
